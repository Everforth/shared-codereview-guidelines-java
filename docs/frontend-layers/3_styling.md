## 表示・スタイリング層（データ表示、UI/UX 等）

### 必須チェック項目：

- **データ表示の実装**:
  - ストアから取得したデータの表示方法が適切か
  - 表示用のcomputedプロパティが適切に使われているか（複雑なビジネスロジックは含まない）
  - v-for、v-if などのディレクティブの使い方が適切か
- **スタイリングの一貫性**: Quasar のデフォルトスタイルを活用し、必要最小限のカスタマイズ
- **ユーザビリティ**: UI要素の配置、ラベルテキストの分かりやすさ

### パフォーマンス考慮点：

- 適切なデータバインディング（v-model の正しい使用）
- 不要な再レンダリングの回避

### 設計・命名規則：

- Scoped CSS の使用
- デザイントークンの活用（`utils/enum.ts` のカラーパレット）
- 意図を表す命名（`five` ではなく `recent`）

### CSS/Sass スタイリング規約：

- **Sassネスティング構文**: `&` を使用した擬似クラス・修飾子の記述を標準とする
- **階層構造**: コンポーネント構造に対応した階層的なCSS記述
- **スコープ管理**: Scoped CSSを使用し、クラス名の衝突を回避

### UI/UX 改善指針：

| ❌ Bad                   | ✅ Good                                     |
| ------------------------ | ------------------------------------------- |
| 'サービスを一時的に停止' | 'AIサービスに障害が発生中'                  |
| `is〜`                   | `can〜` （boolean変数の意味に応じて適切に） |

### レビュー対象外

以下は表示・スタイリング層ではなく、他の層でレビューすること：

- **型定義、ストアの基本実装** → モデリング層で扱う
- **ストアの実装詳細（actions/state/getters）** → モデリング層で扱う
- **ページ構造の基本設計** → プロトタイピング層で扱う
- **computed内の複雑なデータ加工・計算ロジック** → データ処理層で扱う
- **データのCRUD操作**（フォーム送信、削除確認等）→ データ処理層で扱う
- **エラーハンドリングのロジック** → データ処理層で扱う
- **フォーム送信処理** → データ処理層で扱う
- **複雑なビジネスロジック** → データ処理層で扱う
