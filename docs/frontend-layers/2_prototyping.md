## プロトタイピング層（ページコンポーネント、ルーティング 等）

### 必須チェック項目：

- **ページコンポーネント設計**:
  - ファイル命名規則に従っているか（`{Entity}ListView.vue`, `{Entity}DetailView.vue`）
  - 適度な肥大化を許容し、単一責任の原則に従いすぎてファイルが増えることを避けているか
- **ルーティング設計**: 一貫性のあるURL構造とページ遷移設計か
- **基本UI構造**: Quasar を活用した基本的なレイアウトが実現できているか
- **onMountedの必要性**: `onMounted`を使っている場合、DOM操作など本当にマウント後でなければならない理由があるか。`<script setup>`のトップレベルコードはそのまま実行されるため、単純なStore呼び出しは`onMounted`でラップする必要がない

### Vue SFC設計思想：

- **適度な肥大化を許容**: ページコンポーネントはある程度肥大化してよく、分離は最小限に留める
- **表示目的のみの分割禁止**: 単純な表示目的だけでのコンポーネント分割は避ける
- **結合度を減らし凝集度を高める**: 「一緒に変更されるものは、近くに配置すべき」

### Vueコンポーネント実装規約：

- **script setup使用**: `<script setup lang="ts">` を標準とする
- **データフェッチのタイミング**: script setupのトップレベルで直接呼び出し、`onMounted`は不要（DOM操作が必要な場合のみ使用）
- **Store state参照**: `computed()`でラップしてリアクティビティを確保
- **フェッチ前の状態クリア**: 新しいデータをフェッチする前に既存のstateをクリアする

### 画面遷移パターン：

#### ダイアログを使用するケース

- 単純なCRUD操作（新規作成、編集、削除）
- 簡易的な設定変更（ステータス変更、表示順変更）
- 確認を必要とする操作

#### 別画面を使用するケース

- 複雑なフォーム（多数の入力項目、依存関係がある場合）
- 複数のステップを必要とする操作
- 複数のリソースを同時に操作する場合

### パフォーマンス考慮点：

- ページ遷移の導線設計
- コンポーネントの適切な分割（表示目的のみでの分割は避ける）

### 設計・命名規則：

- 一覧画面: `/[entity]` （例: `/games`, `/products`）
- 詳細画面: `/[entity]/:id` （例: `/games/:uuid`, `/products/:id`）
- ファイル名はパスカルケース、ディレクトリ名はケバブケース

### レビュー対象外：

以下はプロトタイピング層ではなく、他の層でレビューすること：

- **Store側のデータ取得ロジック** → モデリング層で扱う
- **型定義、ストア実装** → モデリング層で扱う
- **詳細なスタイリング**（CSS設計）→ 表示・スタイリング層で扱う
- **データのCRUD操作**（作成・更新・削除処理）→ データ処理層で扱う
- **フォームバリデーション** → データ処理層で扱う
- **onMounted内の具体的な処理内容** → データ処理層で扱う
